# Task Manager CLI — Technical Design

## 1. Data Models

### Task

| Property      | Type     | Required | Validation Rules                                      |
|---------------|----------|----------|-------------------------------------------------------|
| `id`          | `string` | Yes      | Generated by `crypto.randomUUID()`; never user-supplied |
| `title`       | `string` | Yes      | Non-empty after `trim()`                              |
| `description` | `string` | No       | Must be a string when provided; defaults to `""`     |
| `status`      | `string` | Yes      | Enum: `todo` \| `in-progress` \| `done`; case-sensitive; defaults to `todo` |
| `priority`    | `string` | Yes      | Enum: `low` \| `medium` \| `high`; case-sensitive; defaults to `medium` |
| `createdAt`   | `string` | Yes      | ISO 8601; set once at creation via `new Date().toISOString()` |
| `updatedAt`   | `string` | Yes      | ISO 8601; refreshed on every update via `new Date().toISOString()` |

### Validation Rules per Property

| Property | Checked by | Rule | Error message returned |
|---|---|---|---|
| `id` | `tasks.js` (`getTask`) | Must match an existing task's UUID; never accepted as user input on `add` | `'Task with id "<value>" not found'` |
| `title` | `validate.js` | Must be a non-empty string after `trim()`; required on `add` | `'title is required and must not be blank'` |
| `description` | `validate.js` | Must be a string when provided; empty string is allowed | `'description must be a string'` |
| `status` | `validate.js` | Must be exactly one of `todo`, `in-progress`, `done`; case-sensitive | `'status must be one of: todo, in-progress, done'` |
| `priority` | `validate.js` | Must be exactly one of `low`, `medium`, `high`; case-sensitive | `'priority must be one of: low, medium, high'` |
| `createdAt` | `tasks.js` | Auto-set; never accepted from user input; not modifiable via `updateTask` | — |
| `updatedAt` | `tasks.js` | Auto-set; never accepted from user input; set to `new Date().toISOString()` on every successful update | — |

**Rules:**
- `validate(field, value)` returns `null` on success, or a human-readable error string on failure — it never throws.
- Properties `createdAt`, `updatedAt`, and `id` are not passed to `validate()`; they are managed internally by `tasks.js`.
- On `update`, only the fields explicitly passed by the caller are validated and merged; omitted fields retain their current values.
- Default values (`status → 'todo'`, `priority → 'medium'`, `description → ''`) are applied in `createTask` before validation runs.

---

**In-memory representation:**

```js
/** @type {Task[]} */
const tasks = [];
```

No serialisation or persistence is performed.

```
src/
  index.js      # CLI entry point: parses process.argv and dispatches commands

  filter.js     # Filter and sort pure utility functions

  validate.js   # Input validation helpers; never throws
```

## 3. Module Responsibilities

### `src/tasks.js`

**Exports:**

| Function | Signature | Description |
|---|---|---|
| `createTask` | `(fields) => Task` | Validates required fields, assigns `id`/timestamps, appends to store |
| `getAllTasks` | `() => Task[]` | Returns a shallow copy of the tasks array |
| `getTask` | `(id) => Task \| undefined` | Returns the task matching `id`, or `undefined` |
| `updateTask` | `(id, fields) => Task` | Merges `fields` into the task and refreshes `updatedAt`; throws if not found |
| `deleteTask` | `(id) => void` | Removes the task by `id`; throws if not found |

**Dependencies:** `node:crypto`

---

### `src/validate.js`

**Exports:**

| Function | Signature | Description |
|---|---|---|
| `validate` | `(field, value) => string \| null` | Returns `null` on success, or a human-readable error string on failure |

Allowed field names: `title`, `description`, `status`, `priority`, `id`, `sort`.

Callers are responsible for checking the return value and calling `process.exit(1)`.

**Dependencies:** none

---

### `src/filter.js`

**Exports:**

| Function | Signature | Description |
|---|---|---|
| `filterByStatus` | `(tasks, status) => Task[]` | Returns tasks matching the given status |
| `filterByPriority` | `(tasks, priority) => Task[]` | Returns tasks matching the given priority |
| `sortTasks` | `(tasks, field) => Task[]` | Sorts by `createdAt` (ascending) or `priority` (`high` > `medium` > `low`) |

All functions are pure — they do not mutate the input array.

**Dependencies:** none

---

### `src/display.js`

**Exports:**

| Function | Signature | Description |
|---|---|---|
| `printTable` | `(tasks) => void` | Prints tasks as a fixed-width plain-text table to `stdout` |
| `printTask` | `(task) => void` | Prints a single task's full detail view to `stdout` |
| `printSuccess` | `(message) => void` | Prints a success message to `stdout` |
| `printError` | `(message) => void` | Prints `Error: <message>` to `stderr` |

**Dependencies:** none

---

### `src/index.js`

**Exports:** nothing (side-effect entry point)

Parses `process.argv` for the following commands and flags:

| Command | Required flags | Optional flags |
|---|---|---|
| `add` | `--title` | `--description`, `--status`, `--priority` |
| `list` | — | `--status`, `--priority`, `--sort` |
| `update` | `--id` | `--title`, `--description`, `--status`, `--priority` |
| `delete` | `--id` | — |

**Dependency graph:**

```
index.js
  ├── tasks.js
  │     └── node:crypto
  ├── filter.js
  ├── display.js
  └── validate.js
```

## 4. Error Handling Strategy

### Error types and origin

| Error type | Where thrown | Handling |
|---|---|---|
| Missing required field | `src/validate.js` → `validate()` returns non-null | Caller calls `printError` + `process.exit(1)` |
| Invalid enum value (`status`, `priority`, `sort`) | `src/validate.js` → `validate()` returns non-null | Caller calls `printError` + `process.exit(1)` |
| Task not found (unknown `id`) | `src/tasks.js` → `updateTask` / `deleteTask` throw `Error` | Caught in `src/index.js`; calls `printError` + `process.exit(1)` |
| Unknown CLI command or flag | `src/index.js` | Calls `printError` + `process.exit(1)` |
| Unexpected programming error | Anywhere | Not caught; propagates to produce a stack trace |

### Rules

- Only `src/index.js` calls `process.exit()`.
- `src/tasks.js` throws `Error` objects with descriptive messages; it never calls `process.exit()`.
- `src/validate.js` never throws; it only returns strings.
- `src/display.js` writes errors to `stderr` via `printError`; all other output goes to `stdout`.
- No `try/catch` around in-memory array operations — they cannot throw unless there is a programmer bug, which should surface as a stack trace.

### Exit codes

| Code | Meaning |
|------|---------|
| `0`  | Success |
| `1`  | User or validation error |
| `2`  | Unrecoverable internal error (reserved) |
